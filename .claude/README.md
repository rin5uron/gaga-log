## 📝 プロセスログの運用ルール

### プロセスログの場所
`docs/PROCESS-LOG.md`

### プロセスログに記録すること

1. **実装・修正の記録**
   - 機能追加
   - バグ修正
   - デザイン変更
   - 設定変更（Analytics、Search Consoleなど）
   - エラーと解決方法

2. **フェーズの進捗**
   - マイルストーン達成
   - AdSense申請チェックリストの更新
   - 記事数の更新

### 記録のフォーマット

```markdown
## YYYY年MM月DD日：タイトル

### 実施内容

#### 1. 機能名

**問題点**:
- 問題の説明

**解決方法**:
- 解決策の説明
- コード例（必要に応じて）

**実装場所**: `path/to/file.tsx`
```

### 記録のタイミング

- 作業完了後、すぐに記録する
- 複数の作業をまとめて記録してもOK（同じ日なら）
- 最新の記録を上部に追加

### 重要な原則

1. **技術的な詳細を残す**
   - エラーメッセージ
   - 解決に至った理由
   - 参考にしたドキュメント

2. **未来の自分が読んでわかるように**
   - なぜその実装をしたのか
   - どんな問題があったのか
   - どう解決したのか

3. **他のプロジェクトでも使えるように**
   - 汎用的な知見は詳しく記録
   - Next.js、Vercel、Google関連の設定は特に

## 🎯 他のプロジェクトでも同じ運用を

このPROCESS-LOG.mdの運用ルールは、全てのリポジトリで共通化します。

### 新しいプロジェクトを始める時

1. `docs/PROCESS-LOG.md`を作成
2. プロジェクト概要、フェーズ管理、実装記録のセクションを設ける
3. 作業のたびに記録する

### メリット

- プロジェクトの歴史が残る
- 過去の実装を振り返りやすい
- トラブルシューティングが簡単
- 他のプロジェクトに知見を活かせる
