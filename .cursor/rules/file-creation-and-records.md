# ファイル作成・記録のルール

## 絶対に守ること（提案・案・記録はファイルに書かない）

- **提案・案・ドラフト・一覧・メモ・TODO はすべてチャット（会話）で出す。ファイルには書かない。**
- ユーザーが「ファイルに書いて」「ここに記録して」と明示したときだけ、指定されたファイルに書く。
- **記録・実施内容・issue の確認は CLI（`gh issue list` など）で行う。** 記録用の新規 .md を .github/ や docs/ に作らない。
- **確認結果の一覧・レポート・チェック結果も「記録」に含む。** これらをまとめた新規 .md（例: docs/〇〇-check.md, 〇〇-report.md）は作らない。issue に書くか、チャットで出す。

## 記録・issue 関連

- **ユーザーが issue 番号を言ったら、必ず `gh issue view <番号>` で本文を取得してから作業する。** 詳細は [issue-cli.md](./issue-cli.md)。
- **記録は GitHub の issue で行う。新規の .md ファイルは作らない。**
- 記録を残す依頼があったら、まず **`gh issue list`** で該当 issue の有無を確認する。
- 該当 issue がある場合：
  - **本文（body）**：実施内容・完了条件・エラー記録など、実装に合わせて編集する。
  - **コメント**：追加でよい場合は `gh issue comment` で追記する。
- 該当 issue がない場合のみ、issue を新規作成するかユーザーに確認する。
- **.github/ISSUE_TEMPLATE/issues/ などに「記録用」の新規 .md は作らない。** 記録は GitHub issue 上で行う。

## ファイル作成・編集の事前確認

- **新規ファイルを追加する前に、このルール（file-creation-and-records.md）を必ず読み直す。** 「レポート」「確認結果」「一覧」を .md で残す依頼は、明示されていない限りチャット or issue で完結させる。
- 任意のファイル作成・編集前に、**既存ファイルの有無を必ず確認**する。
- 既存ファイルがある場合：内容を確認し、追加・修正箇所を明示して承認を得る。
- **新規ファイルを作成する場合も、作業前にユーザーに確認を取る。**
- 記録用の新規 .md ファイルは作らず、GitHub issue に記録する。

## 記録のフォーマット

- ユーザーが指定しない限り、既存のデータ・フォーマットを変えない。
- 既存の issue 本文がある場合は、実施内容・完了条件を「実際にやったこと」に合わせて書き換え、きちんと記録する。
