# 基本原則・プロジェクト構造 - gaga-log

## サイトのコンセプト

### なぜ始めたか

ガガのライブに行くために曲を調べていた。和訳サイトを見ながら、「広告が多いし、そういうサイトが減っている」と気づいた。著作権周りで削除されているようだ。

でも**需要はある**と思った。ライブに行くなら、曲の背景やワンフレーズだけでも歌えるようになりたい。そういう媒体が少ないから、**私が作りたい**と思った。

ノートに書くのは今までもやってきたけど、完成しきれなかったり中途半端でどっか行ったり。だから**デジタルに、しかもURLの形として残せる**のは良いと思った。

どうせ作るなら**財産にしたい**。私の人生の財産、そして**お金にしたい**。ドメイン代と労力が報われるくらい。

### サイトの目的

1. **自分が読み返して嬉しい音楽アーカイブを作る**
2. **読者に曲の新しい聴き方・視点を提供する**
3. **Google AdSense収益化**（有用性の高いコンテンツとして評価される）

### Will / Need / Can

| 観点 | 内容 |
|------|------|
| **Will** | 音楽を深掘りすることで、豊かな人生を送る。自分の好きな音楽をもっと深く愛せるようになる |
| **Need** | 音楽を深掘りして、ライブや生活を豊かにしたい人がいる。「調べて終わり」じゃなく「読んで、もっと好きになる」体験が不足 |
| **Can** | 音楽への愛を言葉で記録し、共有できる。表面的な解説ではなく、深掘りした内容を書ける |

## 記事のスタンス

### ✅ 良い記事

- **解釈・考察**: 「この曲は〇〇を表現している」「歌詞の〇〇は〇〇を意味する」
- **客観的事実＋独自の解釈**: データ・インタビューを基にした考察
- **読者が「もっと好きになる」内容**: 表面的な和訳ではなく、深掘りした視点

### ❌ 避ける記事

- **感想文**: 「私はこう感じました」「個人的には...」
- **推測の羅列**: 「〜かもしれない」「〜だと思う」の連続
- **和訳コピペ**: 他サイトからの丸写し

---

## 想定読者

1. **初めて聴いた人**: 曲の背景、基本的な意味、なぜ有名なのか
2. **ファン**: 深い解釈、他曲とのつながり、文化的背景

## UI/UXルール

### トップページ

- **新着記事**: 6個表示
- **人気記事**: 適宜更新
- **カテゴリ別タブ**: Song / Live / Artist

### 記事ページ

- **ヒーロー画像**: 曲のイメージ画像（必須）
- **この記事でわかること**: 2〜3点、具体的に
- **目次**: アコーディオン式、h2/h3表示、参考情報以降は非表示
- **参考情報セクション**: 公式リンク、ストリーミング、関連記事

### デザイン原則

1. **読みやすさ最優先**: 余白、行間、フォントサイズ
2. **モバイルファースト**: スマホ表示を最優先に設計
3. **情報の階層化**: 見出し、カード、セクションで整理
4. **参考情報と本文の分離**: 目次は本文のみ、参考情報は別セクション

---

## AI運用原則

1. **明確化と確認**: ファイル変更や破壊的コマンドの実行前に、計画を伝えて確認を求める
2. **構造への準拠**: 既存のプロジェクト構造、規約、コーディングスタイルに従う。テンプレートがあれば使用する
3. **人間との連携**: 主観的判断や外部の非公開データ（Google Analytics、Search Console）が必要な場合は、必ず人間に情報提供を求める
4. **ツールの使用**: ファイルシステムとのやり取りには利用可能なツールを使用する。ファイル内容を推測しない
