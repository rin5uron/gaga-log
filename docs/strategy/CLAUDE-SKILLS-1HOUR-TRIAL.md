- note風から音楽専門サイト感へ
- どう変更すべきか方向性が定まっていない

**スキルの使い方:**
```
/skill brainstorming

タスク: 「サイトデザインをnote風から音楽専門サイト感に変更したい」
```

**期待される質問例:**

  - B. カラースキーム

- Q2: ターゲットユーザーの印象は？
  - A. プロフェッショナル・洗練
  - B. 温かみ・親しみやすさ
  - C. エッジィ・先進的

- Q3: 参考にしたいサイトは？
  - [実際に答える]

**ゴール:**

### タスク2: SEO記事一括改善（writing-plans）

**現状の課題:**
- 38記事のSEO構造を一括で統一したい
- 「7つのセクション」が揃っていない記事がある
- 一括修正の手順が不明確

**スキルの使い方:**
```
/skill writing-plans

タスク: 「38記事すべてにSEO戦略の7セクション構造を適用する」

参照ドキュメント:
- docs/seo/SEO-STRATEGY.md（7セクション定義）
- content/posts/*.md（対象ファイル）
```

**期待される計画:**
```
# タスク1: 現状分析スクリプト作成（3分）
- 各記事の見出し構造をチェックするスクリプト
- 不足セクションをリストアップ

# タスク2: bad-romance.mdで検証（5分）
- 7セクション適用
- レイアウト確認

# タスク3: 記事を3グループに分類（2分）
- 完璧（そのまま）
- 微修正（1-2セクション追加）
- 大幅修正（3セクション以上追加）

# タスク4: 微修正グループから順次適用（15分）
- 各記事にセクション追加
- git commit（5記事ごと）

# タスク5: ビルド確認（2分）
```

**ゴール:**
- `docs/plans/2026-01-30-seo-structure-unification.md` が生成される
- 作業の優先順位が明確になる
- 2-5分タスクに分解される

**所要時間:** 20分

### タスク3: 既存の問題を整理（systematic-debugging）

**現状の課題:**
- 「新機能で今すぐ入れたいもの」が曖昧
- 優先度がつけられていない
- 問題点が言語化されていない

**スキルの使い方:**
```
/skill systematic-debugging

問題: 「サイトに何を追加すべきか優先順位がつけられない」
```

**期待される分析:**
```
# 1. 症状の特定
- AdSense申請に必要なもの vs 欲しいもの
- ユーザー体験向上 vs SEO向上
- 開発時間（短期 vs 長期）

# 2. 仮説の立案
- 構造化データ（schema.org）→ SEO効果高、開発時間短
- 検索機能 → UX向上、開発時間長
- ダークモード → デザイン改善、開発時間中
- コメント機能 → エンゲージメント、開発時間長

# 3. 検証
- 各機能のAdSense審査への影響
- 実装難易度
- 効果の即効性

# 4. 優先順位決定
1位: 構造化データ（今すぐ）
2位: ダークモード（デザイン見直しと同時）
3位: 検索機能（記事50本超えたら）
4位: コメント機能（保留）
```

**ゴール:**
- 新機能の優先順位が決まる
- 次にやるべきことが明確になる

**所要時間:** 15分
