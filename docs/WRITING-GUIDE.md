## 基本方針

### サイトの目的
- 自分が読み返して嬉しい音楽アーカイブを作る
- 読者に曲の新しい聴き方・視点を提供する
- Google AdSense収益化（有用性の高いコンテンツとして評価される）

### 記事のスタンス
- ✅ **解釈・考察**: 「この曲は〇〇を表現している」「歌詞の〇〇は〇〇を意味する」
- ❌ **感想文**: 「私はこう感じました」「個人的には...」
- ✅ **客観的事実＋独自の解釈**: データ・インタビューを基にした考察
- ❌ **推測の羅列**: 「〜かもしれない」「〜だと思う」の連続

### 想定読者
1. **初めて聴いた人**: 曲の背景、基本的な意味、なぜ有名なのか
2. **ファン**: 深い解釈、他曲とのつながり、文化的背景

## デザイン要素とその意図

### 🎯 この記事でわかること（highlights）

**意図**:
- 読者が「読む価値」を一目で判断できる
- SEOキーワードを含めて検索流入を増やす
- 記事の焦点を明確にする（書く側の指針）

**実装**:
- frontmatterの`highlights`フィールドに2〜3点記載
- 記事上部に自動表示される（`components/ArticleHighlights.tsx`）

**書き方のルール**:
- **2〜3点まで**（多すぎると焦点がぼける）
- **具体的＋問いかけ**: 抽象語を避け、固有名詞を入れる
- **SEOキーワード含む**: 検索されそうな言葉を入れる
- **一文で意味が通じる**: 簡潔に

**良い例**:
```yaml
highlights:
  - ガガとビヨンセの共演背景
  - Paparazziから続く"超大作MV"の考察
```

**悪い例**:
```yaml
highlights:
  - 「Paparazzi」が描く名声の光と影——愛と執着の境界線（長すぎる、抽象的）
  - MVの衝撃的な殺人シーンに隠されたメッセージ（長すぎる）
  - メディア性の光と影（抽象的すぎる）
  - この曲の魅力について（曖昧）
  - ガガの表現力（焦点が不明確）
```

---

### 📖 目次（TableOfContents）

**意図**:
- **記事の構造を一目で理解**できる
- **アコーディオン式**で閉じられる → モバイルでコンテンツを圧迫しない
- **h2/h3両方表示**で詳細な構造を見せる
- **参考情報セクション以降を除外** → 本文と参考資料を明確に分離

**実装の特徴**:
1. **h2はセクション見出し**（リンクなし）
   - 例: 「この曲について」「歌詞のポイント」
2. **h3はクリック可能**（スムーズスクロール）
   - 例: 「"Shallow"という言葉の二重の意味」
3. **スクロール連動**でアクティブな見出しをハイライト
4. **参考情報以降は非表示** → 目次は本文のみ

**なぜアコーディオンにしたのか**:
- モバイル表示で目次が長すぎると、スクロールが必要になる
- 「読者が必要な時だけ開く」UI → 本文に集中できる
- 初回は開いた状態で表示 → 新規読者に構造を見せる

---

### 🔗 参考情報セクション（References）

**意図**:
- **出典を明記** → 信頼性向上、AdSense評価向上
- **存在感を抑える** → 本文とリンク集の違いを明確に
- **ピンポイント引用は本文内**、一覧はここにまとめる

**実装**:
- CSS（`globals.css`）でフォント・色・余白を控えめに
- `.references-section`クラスで自動適用

**ルール**:
1. **本文中にリンク**: 具体的な情報（リリース日、プロデューサー、発言など）
   - 例: 「[2016年10月21日にリリース](Billboard記事)」
2. **Referencesセクション**: 参考にした記事・MV・インタビューの一覧
   - 公式MV
   - インタビュー・発言
   - その他

**情報源の優先順位**:
1. 公式サイト・レーベル（最優先）
2. Billboard、Rolling Stone、NME（音楽メディア、推奨）
3. Apple Music、Spotify（ストリーミングサービス）
4. Discogs、AllMusic（音楽データベース）
5. Wikipedia（最終手段、AdSense的に不利）

## 記事構成とテンプレート

### Frontmatter（必須）

```yaml
---
title: "曲名"
song: "曲名"  # ← トップページ表示に必須
artist: "アーティスト名"
album: "アルバム名"
year: YYYY
date: "YYYY-MM-DD"  # 作成日（初回公開日）
updatedDate: "YYYY-MM-DD"  # 更新日（構成見直し日、初回はdateと同じ）
order: 1
type: "song"
highlights:
  - ハイライト1（SEOキーワード含む、具体的）
  - ハイライト2（読者が知りたい核心）
  - ハイライト3（あれば3つ目）
---
```

**よくある抜け漏れ**:
- ❌ `song`フィールドなし → トップページに表示されない
- ❌ `date`が未来の日付 → 表示順序がおかしくなる
- ❌ `highlights`なし → ハイライトボックスが表示されない
- ❌ `updatedDate`なし → 更新日が表示されない

---

### 日付（date/updatedDate）のルール

#### dateフィールド（作成日）
- **初回公開日を記録**する（記事を最初に書いた日）
- **一度設定したら変更しない**
- 形式: `"YYYY-MM-DD"`（ISO形式）

**例**:
```yaml
date: "2025-12-31"  # 2025年12月31日に初回公開
```

#### updatedDateフィールド（更新日）
- **記事の構成・考察を見直した日**を記録
- **誤字脱字の修正だけでは更新しない**
- **内容の大幅な追記・修正・構成変更をした時**に更新する

**更新すべき場合**:
- ✅ 新しいセクション（H2/H3）を追加
- ✅ 考察・解釈を大幅に書き換え
- ✅ highlightsを変更
- ✅ 参考情報を大幅に追加
- ✅ 構成を変更（セクションの順序変更など）

**更新しない場合**:
- ❌ 誤字脱字の修正
- ❌ リンクの修正
- ❌ 表現の微調整（1〜2文の修正）
- ❌ CSSやレイアウトの変更

**初回公開時**:
```yaml
date: "2025-12-31"
updatedDate: "2025-12-31"  # 初回はdateと同じ
```

**大幅更新時**:
```yaml
date: "2025-12-31"  # 変更しない
updatedDate: "2026-01-20"  # 更新日を新しくする
```

**トップページの日付表示への影響**:
- トップページには`updatedDate`が表示される
- 記事を更新したら、トップページの日付も自動的に新しくなる
- 読者は「この記事は最近更新されている」と分かる → SEO効果

#### 2. H2セクション（固定、全記事共通）

```markdown
## About <span class="section-subtitle">この曲について</span>
## Lyrics <span class="section-subtitle">歌詞のポイント</span>
## Favorite Lines <span class="section-subtitle">口ずさみたいフレーズ</span>
## Music Video <span class="section-subtitle">MV見どころ</span>
## Analysis <span class="section-subtitle">考察</span>
## References <span class="section-subtitle">参考情報</span>
```

**各セクションの違い**:
- **About**: 曲の背景、リリース情報、アーティストの挑戦
- **Lyrics**: 歌詞の意味、テーマ、構造の「解説・考察」
- **Favorite Lines**: 具体的なフレーズの「引用」と実用的なポイント
- **Music Video**: MVの特徴的なシーン、ライブパフォーマンス
- **Analysis**: 曲の本質、文化的背景、他曲とのつながり
- **References**: 参考情報（公式MV、インタビュー、その他）

---

#### 3. H3見出し（基本2つずつ、具体的に）

**ルール**:
- **基本2つずつ**（必要に応じて3つまで）
- **具体的・状況が浮かぶ見出し**
- **固定テンプレートではない** → 曲に合わせて変える

**良い例**（Aboutセクション）:
```markdown
### 映画『アリー／スター誕生』のクライマックスを彩る一曲
### ガガとクーパー、二人の奇跡の共演
```

**良い例**（Lyricsセクション）:
```markdown
### "Shallow"という言葉の二重の意味
### "In the shallow" と "I'm falling"——繰り返される2つのフレーズ
```

**悪い例**:
```markdown
### 曲の背景（抽象的）
### 歌詞について（曖昧）
### 感想（内容が不明確）
```

---

#### 4. 考察の締め（私の感想2〜4行）

**ルール**:
- Analysisセクションの最後のh3のあと、**2〜4行**で感想を入れる
- 解釈・考察の延長として短く書く
- 感情的な感想ではなく、「この曲をどう受け取っているか」を書く

**例**:
```markdown
### この曲が今も聞かれ続けている理由

[客観的事実＋曲の本質＋普遍的テーマ]

この曲を聴くたびに、私は「浅瀬にいても大丈夫」と思える。深い海に飛び込む勇気がなくても、今ここで生きていていいんだと教えてくれる。それが「Shallow」が愛され続ける理由だと思う。
```

## 執筆ルール

### リンクのルール

#### 基本方針
- **外部リンク（音楽メディア・公式など）**: 本文中の該当文言に直接リンク
  - 例: 「[ガガと共にコンセプトを手掛けた](URL)」
  - ❌ 「ガガと共にコンセプトを手掛けた（[IMDb](URL)）」← 括弧で出さない
- **内部リンク（自サイト）**: 初出のみリンク、2回目以降はリンクなし
  - 例: 初出「[Lady Gaga](/artists/lady-gaga)」→ 2回目以降「ガガ」
- **参考文献欄**: 事実確認・出典の一覧。本文でリンクしたものもここにまとめてOK

#### 自己言及のルール
- **そのページの該当アーティストと該当曲はリンクなし**
- アーティストページでは、そのアーティストの曲へのリンクは付けない
- 記事ページでは、その記事の曲名へのリンクは付けない
- コラボ曲の場合、参加している全アーティストに個別にリンクを付ける

---

### アーティスト名・曲名の表記ルール

| 場所 | 表記 | 例 |
|------|------|-----|
| Frontmatter | 英語（正式名称） | `artist: "Lady Gaga"` |
| 見出し | 英語（正式名称） | `## About` |
| 本文中（初出） | カタカナ + 英語 | 「レディー・ガガ（Lady Gaga）は...」 |
| 本文中（2回目以降） | カタカナ | 「ガガは...」 |
| リンクテキスト | 英語（正式名称） | `[Lady Gaga](/artists/lady-gaga)` |

---

### 引用・装飾のルール

#### 歌詞引用（重要）
**基本方針**:
- **歌詞の直接引用は最小限に**（著作権・AdSense審査対策）
- **フレーズのみ**に留める（1〜2行まで）
- **長文引用は絶対に避ける**（著作権侵害、AdSense不承認リスク）

**引用の書き方**:
```markdown
> I'm on the edge of glory
```
- **帯なし、背景なし**（`<blockquote>`は左余白・イタリックのみ）
- シンプルなインデントで表示

**引用を使う条件**:
- Favorite Linesセクション: 覚えたいフレーズを1〜2個紹介
- Lyricsセクション: 特定のフレーズを解説する時のみ
- **それ以外は引用なし**で、歌詞の内容を自分の言葉で説明

**AdSenseに引っかからないために**:
- ❌ 歌詞を丸ごとコピペ（著作権侵害）
- ❌ 歌詞を複数段落引用（オリジナルコンテンツが少ない）
- ✅ フレーズのみ引用＋自分の言葉で解説
- ✅ 歌詞の意味・背景を自分の言葉で説明

**例（良い）**:
```markdown
### "Shallow"という言葉の二重の意味

> Tell me something, boy / Aren't you tired tryin' to fill that void?

この冒頭のフレーズは、表面的な関係性への疑問を投げかけている。「その空虚を埋めようとすることに疲れていないか？」という問いは、浅い関係（shallow）に留まることへの問いかけだ。
```

**例（悪い）**:
```markdown
## Lyrics

歌詞全文を引用...
（複数段落にわたる引用）
```

#### 太字・流し読み
- **太字だけ追っても内容が理解できる**ようにする
- 見出し＋太字でメリハリをつける

### セクションの文量目安

| セクション | 目安 | 内容 |
|----------|------|------|
| About | 2〜3段落 | 客観的情報＋主観的感想 |
| Lyrics | 3〜5段落 | 歌詞のポイントを複数説明 |
| Favorite Lines | 2〜3段落 | 覚えたいフレーズを紹介 |
| Music Video | 2〜3段落 | MVの見どころ |
| Analysis | 3〜5段落 | 曲の本質、文化的背景 |
| 考察の締め | 2〜4行 | 私の感想 |

**チェック方法**:
1. 読みやすさ: 1段落が長すぎないか（3〜5文程度）
2. 情報量: 各セクションに十分な情報があるか
3. バランス: 各セクションの文量が均等か

## AdSense対策

### 有用性の低いコンテンツ評価を避けるために

#### ✅ やるべきこと
1. **文字数**: 2,000文字以上（推奨8,000文字以上）
2. **独自の視点**: 客観的事実＋独自の解釈・考察
3. **出典明記**: 信頼できる音楽メディアへのリンク
4. **視覚的な変化**: まとめ枠、引用枠、見出しでメリハリ
5. **流し読みしやすい**: 太字、箇条書き、短い段落

#### ❌ 避けるべきこと
1. **歌詞の直接引用を多用** → 著作権・オリジナリティの問題
2. **リンク集のような記事** → 独自コンテンツが少ない
3. **感想文** → 「私はこう感じた」だけでは価値が低い
4. **文字ばかりの記事** → 読みにくい、離脱率が高い
5. **Wikipedia頼み** → 専門性が低いと見なされる

### 広告配置のルール

**現在の配置**:
- 記事上部: ハイライト・目次の上
- 記事中間: 目次の下
- 記事下部: 本文の下、関連記事の上

**余白の調整**:
- 目次・参考文献まわりの広告余白: `my-0`（コンテンツと広告を密着させすぎない）
- ハイライトと目次の余白: `mb-0`（コンパクトに）

## トラブルシューティング

### トップページに表示されない
- `song`フィールドがあるか確認
- `date`が未来の日付になっていないか確認
- 開発サーバーを再起動（`rm -rf .next && npm run dev`）

### ハイライトが表示されない
- YAMLの文法エラーがないか確認
- ダブルクォートの中にダブルクォートがないか確認

### 目次が表示されない
- H2/H3見出しが正しく書かれているか確認
- `<span class="section-subtitle">` がH2に含まれているか確認

### 広告が表示されない
- AdSense承認前は表示されない（テスト広告が出る場合もある）
- ブラウザの広告ブロッカーをオフにして確認
- 開発者ツールで`<ins class="adsbygoogle">`が存在するか確認
