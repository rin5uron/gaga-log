# Issue管理 - gaga-log

**記録の運用**: 実施内容・完了報告は **GitHub issue** の本文またはコメントに書く。編集は `gh issue edit` / `gh issue comment` を使用。手順は `.cursor/rules/file-creation-and-records.md` と `.cursor/rules/issue-cli.md` を参照。

**書き方**: Issue やコメントは、このサイト・依頼者を知らない人が読んでもわかるように書く。ファイルパス（例: `content/posts/judas.md`）だけにせず、「Lady Gagaの曲『Judas』の記事」のように内容が伝わる表現を添える。

---

## 基本原則

**アイデア・作業履歴・進捗は全てIssueで管理**

- ドキュメントは `docs/` に集約（これ以上増やさない）
- それ以外の情報（アイデア、改善案、作業履歴）は GitHub Issue で管理
- Issue コメントで時系列の記録を残す

---

## タイトル形式

**形式**: `【カテゴリ】対象 - 作業内容`

| カテゴリ | 使う場面 |
|----------|---------|
| `【リライト】` | 記事の書き直し・テンプレ準拠 |
| `【新記事】` | 新規記事作成 |
| `【SEO】` | SEO改善・CTR改善 |
| `【AdSense】` | AdSense審査・収益化対応 |
| `【設計】` | ルール・テンプレ・設計書 |
| `【デザイン】` | UI/UXデザイン |
| `【バグ】` | 不具合修正 |
| `【インフラ】` | サイト構造・技術実装 |
| `【学習】` | Next.js/Tailwind学習 |
| `【記録】` | 作業メモ・実装記録 |

**例**:
- `【リライト】Judas - テンプレ準拠・考察追加`
- `【リライト】Alejandro - H2追加・文体修正`
- `【SEO】Alejandro - CTR改善`
- `【AdSense】全記事 - H2構造変更`
- `【設計】リライトルール`
- `【バグ】Vercelビルド失敗 - layout.tsx img属性`

## Issueクローズのタイミング

- チェックリストがすべて完了したらクローズ
- 新たな問題が発生したら新しいIssueを作成（既存issueに追記しない）

---

## 作業の流れ

### リライト作業（#22・#30・個別issueの3点セット）
1. **#22**（全体方針）・**#30**（リライトルール）を読み込む
2. 対象記事のissueを開き、チェックリストに沿って作業
3. H2変更・セクション順序修正・frontmatter（theme）追加を同時に行う
4. 完了後、issueコメントに記録してクローズ

### 新規記事作成
1. `/new-article` 実行
2. Issue作成（`【新記事】曲名`、ラベル: `content`）
3. 公開後クローズ

### SEO改善
1. `/seo-check` 実行
2. Issue作成（`【SEO】曲名 - 作業内容`、ラベル: `seo,content`）
3. 完了後クローズ

### バグ修正
1. Issue作成（`【バグ】内容`、ラベル: `bug`）
2. 修正完了後クローズ

### 記録・確認作業
1. Issue作成（`【記録】対象 - 作業内容`、ラベル: `content` など該当）
2. 本文に背景・内容・結果を記載
3. 完了後コメントに日付を書いてクローズ

## 記録用Issueの例（コピペ用）

**※ 記録は GitHub issue に残す。以下は issue の本文用テンプレ（コピペ用）。新規のときは `gh issue create --body-file ファイル名`、既存のときは `gh issue edit <番号> --body` で更新する。**
